## 一、架构

### 1.1 核心架构 -- 分层+循环+自适应

Bio-Framework 的架构不是单一的线性流水线，而是一个五层结构，从静态知识到动态执行，
层层递进，每一层都有明确的职责边界：

**核心设计理念**：框架层和知识层是静态的（随框架版本发布），配置层和执行层是
AI 在运行时动态生成的。这意味着框架的所有"智慧"都编码在静态文件中，而 AI 只
需要按照这些规范去执行 -- 降低了 AI 自由发挥导致错误的概率。

### 1.2 三层资源优先级

Bio-Framework 使用三层配置覆盖机制，更具体的配置覆盖更通用的配置：

```
优先级：高 ──────────────────────────────────────────── 低

┌──────────────────┐    ┌──────────────────┐    ┌──────────────────┐
│  Project 层       │    │  User Global 层   │    │  Plugin 层       │
│  (.claude/)       │ >> │  (~/.claude/      │ >> │  (framework/)    │
│                   │    │   bio-framework/) │    │                  │
├──────────────────┤    ├──────────────────┤    ├──────────────────┤
│ 项目特定配置       │    │ 用户个人偏好       │    │ 框架默认值       │
│ 项目规则           │    │ 全局规则           │    │ 标准流水线       │
│ 项目知识库         │    │ 用户知识库         │    │ 知识库           │
└──────────────────┘    └──────────────────┘    └──────────────────┘
```

**合并策略对照表**：

| 资源类型 | 合并策略 | 说明 |
|----------|----------|------|
| 参数值 (rules) | Override | 高层直接覆盖低层同名参数 |
| 流水线步骤 (pipelines) | Override | 项目层可完全替换标准流水线 |
| 错误经验 (errors_memory) | Union | 三层错误经验合并，不覆盖 |
| 期刊配置 (journals) | Override | 项目层可定制期刊要求 |
| 图形类型 (figure_types) | Union | 项目层可追加自定义图形类型 |
| 决策节点 (decision_nodes) | Override | 项目层可重定义决策阈值 |

**配置解析示例** -- 聚类分辨率参数的解析流程：

```
查找 clustering.resolution:

  1. 检查 Project (.claude/rules/project_rules.yml)
     -> 找到 resolution: 1.2
     -> 使用 1.2 [命中，停止搜索]

  如果 Project 层未找到:
  2. 检查 User Global (~/.claude/rules/user_rules.yml)
     -> 找到 resolution: 0.6
     -> 使用 0.6 [命中，停止搜索]

  如果 User Global 层也未找到:
  3. 检查 Plugin (framework/rules/default_rules.yml)
     -> 找到 resolution: 0.8
     -> 使用 0.8 [框架默认值]
```

### 1.3 ORCHESTRATOR 子模块架构

ORCHESTRATOR.md 是框架的核心编排器（398 行），但它不承载具体逻辑 --
所有实际工作委托给 13 个专职子模块：

```
ORCHESTRATOR.md (398 行 -- 核心路由与调度)
│
├── [执行控制]
│   ├── execution_steps.md ──────── 执行步骤规范 (Step 0-6 全部步骤定义)
│   ├── checkpoint_recovery.md ──── 检查点恢复 (断点续做 + 状态重建)
│   └── thinking_dispatch.md ────── 思维模式分配 (四级思维路由逻辑)
│
├── [分析阶段]
│   ├── data_quality_review.md ──── Step 1.5 数据质量深度审计 [v1.0新增]
│   ├── reflection_exploration.md ─ Step 2 反思与探索循环
│   ├── research_summary.md ─────── Step 3 研究总结与图表规划
│   ├── publication_figures.md ──── Step 4 出版质量图表 + 视觉审查
│   ├── final_report.md ─────────── Step 5 技术报告
│   └── manuscript_drafting.md ──── Step 6 SCI 稿件写作
│
├── [合规与质量]
│   └── pipeline_compliance.md ──── 管道合规检查 (标准流程验证)
│
├── [课题管理]
│   ├── NEW_TOPIC_CLEANUP.md ────── 新主题环境清理
│   └── topic_change_rules.md ──── 主题变更规则与安全守卫
│
├── [模型路由]
│   └── claude_router.md ────────── 模型选择与路由策略
│
└── [参考]
    └── integration_guide.md ────── 集成指南
```

**子模块总规模**：13 个子模块，合计 10,318 行。
加上 ORCHESTRATOR.md 核心的 398 行，整个编排系统超过 10,700 行。

这种拆分设计的核心原因：Claude 的上下文窗口有限。如果将全部逻辑塞进单个文件，
AI 在处理时必须加载完整的 10,500 行内容。子模块化后，ORCHESTRATOR.md 只需
在特定步骤按需加载对应子模块，显著降低了上下文消耗。

### 关键工作流特性

**动态 Phase 规划** -- Phase 的数量和内容不是硬编码的。Step 0 会根据：
- 组学类型（scRNA-seq vs Bulk RNA-seq vs 代谢组...）
- 数据规模（样本数、细胞数）
- 用户目标（探索性分析 vs 假设驱动）
- 标准流水线定义（pipelines/*.yml）

动态生成 3-8 个 Phase。例如：

```
scRNA-seq 典型: 4-5 个 Phase
Bulk RNA-seq:   3-4 个 Phase
代谢组:         3 个 Phase
多组学整合:     6-8 个 Phase
```

**暂停点设计** -- 工作流中有 3 个用户决策暂停点：

```
暂停点          位置         目的
────────────────────────────────────────────────
探索门          Step 2b      用户决定是否深化分析
用户视觉审查    Step 4f      用户确认图表质量
质量审计异常    Step 1.5     CRITICAL 级别暂停
```

这些暂停点确保 AI 不会在关键决策上自作主张。

Bio-Framework 定义了四级思考深度，每个步骤/场景有明确的思考模式分配：

### 4.4 错误记忆与自动恢复

Bio-Framework 的错误处理不是简单的 try-catch，而是一个三层经验驱动系统：

```
错误发生
    │
    v
┌────────────────────────────────────────────────────────────────┐
│ 第一层: 框架知识库 (COMMON_ERRORS.yml, 20KB)                    │
│                                                                 │
│ 包含数百条预编码的错误模式:                                       │
│ - R 包安装失败 (BiocManager/remotes/CRAN 差异)                  │
│ - Seurat 版本兼容性问题                                          │
│ - 内存溢出 (大数据集 + 高分辨率聚类)                              │
│ - 常见数据格式错误 (10X vs h5ad vs CSV)                          │
│                                                                 │
│ 匹配方式: 错误信息模式匹配 -> 对应解决方案                        │
└───────────────┬────────────────────────────────────────────────┘
                │ 未匹配
                v
┌────────────────────────────────────────────────────────────────┐
│ 第二层: 用户经验库 (errors_memory.yml, 运行时累积)               │
│                                                                 │
│ 每次成功解决错误后自动记录:                                       │
│ ┌────────────────────────────────────────────────────┐         │
│ │ errors_memory.yml:                                  │         │
│ │   - error_pattern: "cannot open connection"         │         │
│ │     context: "读取 10X 数据"                         │         │
│ │     solution: "检查文件路径，确认 barcodes/genes/    │         │
│ │                matrix 三个文件齐全"                   │         │
│ │     resolved_at: "2026-03-01"                       │         │
│ │     success: true                                   │         │
│ └────────────────────────────────────────────────────┘         │
│                                                                 │
│ 同一错误再次出现时，直接应用历史解决方案                           │
└───────────────┬────────────────────────────────────────────────┘
                │ 未匹配
                v
┌────────────────────────────────────────────────────────────────┐
│ 第三层: AI 自主推理                                              │
│                                                                 │
│ 使用 DEEP 或 ULTRATHINK 模式分析:                                │
│ - 错误上下文 (哪一步、什么数据、什么参数)                         │
│ - 可能原因推理                                                   │
│ - 尝试解决方案                                                   │
│ - 成功后写入 errors_memory.yml (供后续复用)                       │
│                                                                 │
└────────────────────────────────────────────────────────────────┘
```

**关键设计**: 错误经验是**累积的**，不是覆盖的。三层经验通过 Union 策略合并，
这意味着项目层的经验不会覆盖框架层的经验，而是补充。

### 4.5 决策节点系统

框架在工作流的关键分岔点定义了形式化的决策节点，避免 AI 凭"直觉"做出
不可逆的决策：

```yaml
# decision_nodes.md 中的决策节点定义示例 (简化)
decision_nodes:
  overclustering_check:
    trigger: "聚类结果中出现可能的过度聚类"
    assessment:
      - metric: "cluster_count_vs_cells"
        threshold: "n_clusters > n_cells / 100"
        note: "简化公式，实际需结合数据特征"
      - metric: "silhouette_score"
        threshold: "< 0.1 for multiple clusters"
      - metric: "marker_gene_overlap"
        threshold: "> 50% shared markers between clusters"
    response_levels:
      low:    "记录警告，继续分析"
      medium: "降低分辨率参数，重新聚类"
      high:   "[HARD_STOP] 暂停，请求用户确认聚类策略"

  sample_identity_mismatch:
    trigger: "样本标签与基因表达模式不一致"
    assessment:
      - check: "性别基因表达 vs 标注性别"
      - check: "物种特异基因 vs 标注物种"
    response_levels:
      confirmed: "[CRITICAL] 暂停分析，样本可能被交换"
```

**分级响应机制**：

```
┌──────────┬──────────────────────────────┬──────────────────────┐
│ 级别      │ 动作                          │ 是否暂停              │
├──────────┼──────────────────────────────┼──────────────────────┤
│ LOW      │ 记录到 decision_log.yml       │ 否                   │
│ MEDIUM   │ 自动调整参数 + 记录            │ 否                   │
│ HIGH     │ 暂停执行，展示给用户           │ 是 [HARD_STOP]       │
│ CRITICAL │ 立即停止，等待用户指令         │ 是 [HARD_STOP]       │
└──────────┴──────────────────────────────┴──────────────────────┘
```

### 4.6 自适应能力的技术支撑

Bio-Framework 的自适应不是魔法 -- 它由具体的技术机制支撑：

**Phase 动态调整**：

```
Step 0 (初始规划)          Phase 执行中 (动态调整)
──────────────────         ──────────────────────────

规划 5 个 Phase            Phase 2 发现 3 个批次效应严重
  Phase 1: QC                 -> 动态插入 Phase 2.5: 批次校正
  Phase 2: 降维+聚类          -> 后续 Phase 编号自动调整
  Phase 3: 注释
  Phase 4: DE分析           Phase 3 发现细胞类型少于预期
  Phase 5: 高级分析            -> Phase 5 (高级分析) 简化
                               -> 减少亚群分析深度
```

**参数自适应**：

```
CONFIG_MANAGER 参数解析流程:

  1. 加载三层配置 (Project > User > Plugin)
  2. 检查数据特征:
     - 细胞数 < 1000?  -> 降低 min_features 阈值
     - MT% 分布偏高?   -> 调整 mt_percent 阈值
     - 基因数异常少?    -> 检查数据格式
  3. 应用解析后的参数值
  4. 记录实际使用值到 workflow_state.yml
```

**经验积累**：

```
会话 1: scRNA-seq 分析
  -> 遇到 Seurat v5 兼容性问题
  -> 解决并记录到 learned_solutions.yml

会话 2: 另一个 scRNA-seq 分析
  -> 遇到相同问题
  -> learned_solutions.yml 命中
  -> 直接应用历史解决方案 (无需重新推理)

### 4.7 视觉审查闭环 [v1.0 新增]

claude_router.md 定义了思考模式到 Claude 模型的映射策略，

│                                    │ STANDARD  -> Sonnet │       │

│                                    └────────────────────┘       │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 成本优化策略

```
场景                    思考模式     模型        相对成本
──────────────────────────────────────────────────────
读取文件内容             QUICK       Haiku       $
标准 QC 分析             STANDARD    Sonnet      $$
图表代码生成             DEEP        Sonnet      $$
细胞类型注释             ULTRATHINK  Opus        $$$$
SCI 稿件写作             ULTRATHINK  Opus        $$$$

典型 scRNA-seq 全流程:
  Step 0 (ULTRATHINK):  1 次 Opus 调用
  Phase 1-5:            ~20 次 Sonnet + ~5 次 Opus (关键Phase)
  Step 1.5-6:           ~30 次 Sonnet + ~8 次 Opus
  文件操作:              ~50 次 Haiku

总计: ~60% Sonnet + ~15% Opus + ~25% Haiku
相比全程使用 Opus，成本可显著降低（具体幅度取决于工作负载）

### 安全关键点

| 安全维度 | 保障措施 |
|---------|---------|
| 数据存储 | 所有数据文件保持在用户本地磁盘 |
| 代码执行 | R/Python 脚本在用户本地环境执行 |
| AI 交互 | 仅传输代码文本和摘要统计 |
| 敏感信息 | 患者ID/样本标识不包含在AI交互中 |
| 结果文件 | 图表/报告/稿件存储在本地 project_skills/ |
| 会话隔离 | 不同项目的数据严格隔离 (TOPIC.yml 机制) |

### v1.0.0 的核心设计哲学

```
  早期版本: "让 AI 自由发挥，做生物信息学分析"
       |
       |  问题: AI 容易跳步、遗漏关键检查、做出错误的生物学判断
       |
       v
  v1.0.0: "用显式规则约束 AI，确保分析质量"

  具体体现:
  ┌────────────────────────────────────────────────────────────┐
  │                                                            │
  │  1. 强制 Ultrathink -- 不允许 AI 在关键决策上"抄近路"       │
  │                                                            │
  │  2. 暂停点 -- 关键决策必须经过人类确认                       │
  │                                                            │
  │  3. mandatory 图检查 -- 不能遗漏核心图表                    │
  │                                                            │
  │  4. 五维度质量审计 -- 系统化的数据验证                       │
  │                                                            │
  │  5. 视觉审查闭环 -- AI 必须看自己生成的图                   │
  │                                                            │
  │  6. 三层错误经验 -- 不重复犯同样的错误                      │
  │                                                            │
  │  7. 决策节点量化 -- 用阈值而非直觉做判断                    │
  │                                                            │
  │  8. 检查点恢复 -- 不因中断丢失工作进度                      │
  │                                                            │
  │  核心理念: AI 的创造力用在分析上,                            │
  │           框架的规则用在流程控制上。                          │
  │                                                            │
  └────────────────────────────────────────────────────────────┘
