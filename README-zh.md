# Bio-Framework

> **不是又一个 AI 聊天机器人。是 350 KB+ 结构化知识驱动的分析导航系统。**

---

## 先用人话说

你有一批组学数据（单细胞、空间、蛋白质组……），想发 SCI 论文。

Bio-Framework 让 AI 按照生物信息学领域的最佳实践，帮你完成**从数据质控到出版级图表再到论文初稿**的全部工作。在关键节点（数据质量、统计正确性、期刊格式）设置了硬性门控，防止 AI 犯错或编造数据。

**你不需要自己写 R/Python 代码**——AI 会自动生成、执行、调试。但你需要理解自己的科学问题，并在关键判断点（细胞类型注释、分析方向选择）做出决策。

所有分析**在你的本地环境运行**。原始数据不会上传到任何第三方服务器。

---

## 它到底是什么？

先说清楚它**不是**什么：

- **不是计算引擎** —— 它不替代 R/Python，不替代 Seurat，不替代 DESeq2
- **不是 GPT 提示词** —— 不是那种"你是一个生信专家，请帮我分析 scRNA-seq 数据"的 system prompt
- **不是流水线的竞争者** —— 它不替代 Nextflow/Snakemake 的批量计算能力，而是在分析决策层面提供导航

**Bio-Framework 是分析导航层（Guidance Layer）。**

它是 [Claude Code](https://docs.anthropic.com/en/docs/claude-code/overview) 的一个 Skill 插件。你仍然用 R/Python 环境跑分析，但 Bio-Framework 在每一步告诉 AI：

- 这一步**必须做什么**（mandatory steps 强制检查）
- 这一步**容易犯什么错**（16 种常见错误模式，含代码级修复方案）
- 这一步的输出**必须满足什么标准**（SCI 出版标准、期刊合规要求）

它的核心资产不是代码，是那 **350 KB+ 的结构化知识库**——错误模式、SCI 设计标准、工作流逻辑、期刊合规规则、方法论文档，全部以机器可读的 YAML 格式存储，AI 在执行时实时查询和强制执行。

## 让"湿实验"博士无法拒绝的 Hardcore 卖点

### 1. 拒绝"AI 幻觉"：四级思考深度调度

**痛点：** 现在的 AI 给建议很随性。创建一个文件夹，和鉴定一个细胞类型，用的是同一种逻辑深度。你问它"这个 cluster 是什么细胞"，它扫一眼 marker 就给你答案——没有交叉验证，没有负面证据排除，没有文献比对。

**Bio-Framework 的方案：**

框架内置 **四级思考深度**，根据任务关键性自动调度：

| 思考级别 | 触发场景 | AI 行为 |
|:---|:---|:---|
| **Quick** | 文件操作、格式调整 | 快速执行 |
| **Standard** | 常规分析步骤 | 标准推理 |
| **Deep** | 参数选择、方法决策 | 多方案比较 |
| **Ultrathink** | 9 种关键场景 | 完整推理链：观察→假设→验证→结论 |

**9 种强制 Ultrathink 场景：**

- 细胞类型鉴定与注释
- 数据质量审查（Step 1.5）
- 差异表达结果解读
- 生物学结论推导
- 稿件关键章节撰写（Results / Discussion）
- 数据验证（6e 阶段：每个统计数字追溯到源数据）
- 聚类分辨率选择
- 批次效应评估
- 异常结果判断

在 Ultrathink 模式下，AI **必须**完成完整推理链——不是给你一个答案，而是展示它**怎么得出**这个答案。

---

### 2. 拒绝"带毒数据"：Step 1.5 五维度深度审计

**痛点：** 分析跑了两周，图都画完了，突然发现样本标签搞反了。或者审稿人指出你的批次效应没去干净，所有下游分析全部作废。

**Bio-Framework 的方案：**

在 Step 1（计算分析）完成后、Step 2（解读）开始前，框架**强制插入** Step 1.5 数据质量审计。五个维度，一个都不能跳过：

| 审计维度 | 检查内容 |
|:---|:---|
| **样本身份** | 样本标签是否正确、是否有交叉污染信号 |
| **批次效应** | 批次校正是否充分、残留批次效应量化 |
| **技术质量** | 测序深度、检测基因数、线粒体比例等 QC 指标 |
| **生物学合理性** | 已知 marker 是否出现在预期的细胞群中 |
| **统计假设** | 样本量是否支撑统计结论、多重检验校正是否到位 |

审计结果写入 `quality_report.yml`。**只有全部通过，工作流才会继续。** 这不是建议，是 HARD_STOP 硬性门控。

---

### 3. 拒绝"格式退稿"：内置 6 大顶刊合规引擎

**痛点：** 论文科学内容没问题，但因为分辨率不够、图注格式不对、摘要超字数被 desk reject。

**Bio-Framework 的方案：**

框架内置 **6 种顶刊**的精确合规配置（机器可读 YAML，不是自然语言建议）：

| 期刊 | 关键约束（示例） |
|:---|:---|
| **Nature** | 摘要 ≤150 词，正文 ≤3000 词，引文 ≤50 条，标题避免 "reveals/shows/uncovers" |
| **Science** | 独立稿件格式规范，研究文章/报告分类约束 |
| **Cell Reports** | 图表规范、STAR Methods 必填 |
| **JCI** | 临床研究附加要求，数据共享声明 |
| **Nature Methods** | 方法验证基准、可重复性声明 |
| **PLOS ONE** | 开放获取格式、Data Availability 必填 |

AI 在 Step 6（稿件撰写）时会自动：

- 检查图表分辨率是否达到 **300 DPI**
- 检查字体大小是否符合期刊最小要求
- 检查配色方案是否**色盲友好**
- 检查摘要/正文字数是否超限
- 检查 Data Availability / Code Availability 声明是否完整
- 生成 `submission_readiness.md` 投稿前自查报告

### 4. 拒绝"数据造假"：数据真实性硬性保护

**痛点：** AI 最让科研人员不安的问题——它会不会编造数据？下载失败了会不会悄悄生成模拟数据继续跑？正文里引用的 P 值和实际分析结果对不上？

**Bio-Framework 的方案：**

**三道防线，零容忍：**

**防线一：禁止生成模拟数据（HARD_STOP）**

框架错误库中 F003 为 `severity: critical` 级别。当真实数据缺失、下载失败或解析异常时，AI **必须停下来等待用户决策**，绝不允许"创造性地"生成模拟数据继续工作流。

**防线二：稿件绝对禁止清单**

- 不得捏造实验数据、统计值、P 值或 fold change
- 不得编造文献引用或曲解已发表研究
- 不得夸大结果或隐藏关键局限性
- 信息不足时必须用 `[Ref]` 或 `[XX]` 占位符，不得虚构

**防线三：五步提交前交叉验证（6e—6i）**

| 阶段 | 验证内容 |
|:---|:---|
| **6e Data Validation** | 正文中每个统计数字追溯到分析输出文件（Ultrathink 级别） |
| **6f Claim Review** | 每个结论都有对应的图表/数据支撑 |
| **6g Structural Completeness** | 所有必要章节完整性检查 |
| **6h Cross-File Sync** | 正文数字与图表/补充材料交叉核对（"正文说1234个DEG，表S1必须恰好1234行"） |
| **6i Submission Readiness** | 生成最终投稿准备报告 |

### 5. 六条标准化组学流水线：不只是"知道"，是"强制检查"

**覆盖范围：**

| 组学类型 | 支持的技术平台 |
|:---|:---|
| **单细胞 RNA-seq** | 10x Genomics / Seurat / Scanpy |
| **空间转录组** | Visium / MERFISH / Slide-seq / STARmap / CODEX |
| **批量 RNA-seq** | DESeq2 / edgeR / limma |
| **蛋白质组** | MaxQuant / DIA-NN / TMT / iTRAQ |
| **代谢组** | XCMS / MetaboAnalyst / MZmine |
| **脂质组** | 继承代谢组 + 脂质特异性扩展 |

每条流水线都定义了 **mandatory steps**。AI 执行时会实时对比你的分析是否覆盖了每个必须步骤——跳过 doublet removal？Warning。没做 batch correction？Warning。不是事后补救，是过程中拦截。

**模块化使用**：你不一定要从 Step 0 跑到 Step 6。已经有分析结果了？可以直接从 Step 4（图表生成）或 Step 6（稿件撰写）开始。

### 6. 自学习经验库：用得越多越聪明

**机制：**

```
遇到问题 → 查框架知识库（16种常见错误）
         → 查用户历史经验（learned_solutions.yml）
         → AI 自主推理解决
         → 解决过程 > 2 轮 → 自动保存到用户知识库
```

第一个项目遇到的 Seurat v5 API 变更问题、CellChat 内存溢出解法、R 包编译报错修复——全部自动记录。第二个项目遇到同样问题，直接调用历史方案，`times_reused` 计数器追踪复用次数。

**你积累的不只是数据，还有分析经验。**

---

### 7. 断点续传：跑到一半断了不用从头来

分析跑到 Step 4（图表生成），服务器断了？

```bash
/bio continue
```

系统会：
- 扫描已完成的 `Figure_*.pdf` 文件
- 与 `figure_plan.yml` 对比
- **从第一个缺失的图开始续做，不重新生成已完成的图**

Step 6（稿件写作）有 **21 个独立子步骤**，每个写独立文件。任何一步超时或崩溃，只重试该步骤，已完成的章节不受影响。

---

### 8. SCI 图表知识库：审稿人看什么，它就检查什么

内置 732 行 SCI 设计标准（`sci_design_standards.yml`），不是教你画图，是帮你**避雷**：

- **图表决策树** —— 样本量 n<10 用点图+配对线，n=10-30 用箱线图+抖动点，n>30 用小提琴图
- **错误图表防御** —— 8 种常见错误图表及替代方案（条形图隐藏数据分布→箱线图，双 Y 轴误导→分面图）
- **AI 写作真实性检测** —— 专门识别 AI 生成文字的 7 种特征模式（过度使用"Furthermore/Moreover"、段落长度均匀等），帮你写出真实的专家风格
- **审稿人必查清单** —— 15+ 个审稿人高频质疑项，每项都有具体修复方案

### 9. 三层可定制参数：不是黑箱，是你的个人偏好系统

```
Project .claude/rules/     ← 项目级覆盖（最高优先级）
User ~/.claude/rules/      ← 你的全局偏好
Plugin framework/rules/    ← 框架默认值
```

12 个参数域全部可定制：QC 阈值、标准化方法、降维参数、聚类分辨率、差异表达方法、批次校正、细胞注释、可视化风格、性能配置、输出格式……

你积累的经验（"我的数据集 MT% 通常很低，QC 阈值改为 15%"、"我偏好更高的聚类分辨率 0.6"）可以保存为全局偏好。所有新项目自动继承，单个项目随时覆盖。

## 完整工作流一览

```
Step 0  项目初始化 ─── 自动检测组学类型、创建标准目录结构
  │
Step 1  动态分析 ───── 多阶段计算分析（Phase 1-N），每阶段强制检查 mandatory steps
  │
Step 1.5 数据质量审计 ─ 五维度深度审查，HARD_STOP 门控
  │
Step 2  反思与探索 ─── 科学问题完成度打分（80%/70%），定向探索建议
  │
Step 3  研究总结 ───── 核心发现提炼、图表计划制定
  │
Step 4  出版级图表 ─── SCI 标准输出 + 视觉审查闭环
  │
Step 5  报告生成 ───── 结构化分析报告
  │
Step 6  稿件撰写 ───── 期刊合规 + 五步交叉验证（6e-6i）
```

## 数据安全与隐私

Bio-Framework 完全运行在 Claude Code 本地环境中：

- **数据不离机**：所有原始数据（h5、rds、csv 等）仅由你本地的 R/Python 环境读取和处理
- **本地代码执行**：AI 生成的分析代码在你的机器上运行，不在云端
- **API 通信范围**：Claude Code 作为 AI 服务会与 Anthropic 服务器通信，发送的是代码逻辑和输出摘要——原始数据矩阵不在传输范围内

> **注意**：对于数据安全要求极高的场景（如受 IRB 约束的临床数据），用户应自行评估是否符合其机构的数据管理政策。详情请参阅 [Anthropic 的数据处理政策](https://www.anthropic.com/policies/privacy)。

## 快速上手

**第一步：安装框架**

```bash
./install.sh --global
```

**第二步：在你的数据目录初始化项目**

```bash
cd your_project && bio-init
```

填写生成的 `TOPIC.yml`——用自然语言描述你的研究问题、数据路径、目标期刊。

**第三步：开始分析**

```
/bio start --topic TOPIC.yml
```

框架自动识别组学类型、加载对应流水线、推进全流程。中途可以随时：

```
/bio status      # 查看进度
/bio continue    # 断点续传
/bio help        # 帮助
```

## Bio-Framework 不适合什么

保持诚实：

| 场景 | 原因 | 替代方案 |
|:---|:---|:---|
| 大规模生产级流水线 | 框架是交互式的，不适合每天处理数百样本的测序中心 | Nextflow / Snakemake |
| 底层算法开发 | 框架使用现有工具（Seurat、DESeq2 等），不适合方法学创新 | 直接编程 |
| 纯上游分析 | 基因组组装、比对（alignment）等计算密集型任务 | BWA / GATK / SPAdes |
| 不使用 Claude Code | 框架是 Claude Code 的 Skill 插件，依赖该平台 | — |

> 完整的局限性说明见 [为什么要使用 Bio-Framework — 第六节](why-bio-framework-zh.md#六局限性与注意事项)。
